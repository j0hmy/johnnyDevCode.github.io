移动采编App在线安装方案

发表于 2016-12-07   |  

这周做了一个iOS移动采编App在线安装方案，分享给大家。

如下：
1.首先需要有利用企业账号选择inhouse导出方式打包的ipa文件；
2.需要设计提供两个尺寸的App桌面显示的icon，文件为png，尺寸分别为57x57和512x512；
3.需要一个名字为“manifest.plist”的配置文件，里面的具体见附件中的manifest.plist
4.将以上文件，包括ipa、icon、manifest.plist等存储在咱们自己的服务器上，要求访问URL为https；
5.在web页面中的安装按钮的链接设置为如下格式：
<a href="itms-services://?action=download-manifest&url=https://raw.githubusercontent.com/johnnyDevCode/test/master/manifest.plist">click to download</a>
注意：不符合这种格式的话苹果iOS系统不能识别，就无法安装；蓝色部分为配置文件在服务器上的访问的URL

6.关于升级功能的实现，需要服务器管理安装包的版本，每个版本都需要维护上面的一套ipa和manifest.plist。
7.升级接口中需要返回给客户端安装最新版的URL应该是包含红色部分的固定格式的完整链接，比如:itms-services://?action=download-manifest&url=https://raw.githubusercontent.com/johnnyDevCode/test/master/manifest.plist

AVPlayer Seek精确度到帧

发表于 2016-11-24   |  

AVPlayer是AVFoundation框架下的音视频播放器，可以用来播放音频、视频，并简单封装了播放器的一些控制操作，比如播放、暂停和快进等。本篇文章不对AVPlayer的使用做过多的介绍，主要针对seek（快进、快退）功能的遇到的一些问题进行一些解决方案。

问题点1：AVPlayer当播放视频的时候，进行seek的时候精确度不能精确到小于1秒，也就是不能seek到视频每帧的位置

具体情况是这样的，我们使用AVPlayer进行seek的时候习惯使用以下实例方法

  • (void)seekToTime:(CMTime)time 或者
  • (void)seekToTime:(CMTime)time toleranceBefore:(CMTime)toleranceBefore toleranceAfter:(CMTime)toleranceAfter;
  • (void)seekToTime:(CMTime)time completionHandler:(void (^)(BOOL finished))completionHandler
  • (void)seekToTime:(CMTime)time toleranceBefore:(CMTime)toleranceBefore toleranceAfter:(CMTime)toleranceAfter completionHandler:(void (^)(BOOL finished))completionHandler

后面两个方法就是在前两个的基础上增加了操作完成之后的回调代码块,关键在创建CMTime这个参数上
但是当我们使用seekToTime进行seek的时候发现视频的预览只是每个1秒才变换一个画面，而不是像系统的AVPlayerViewController中拖动进度条那么的流畅。

后来通过stack overflow上找到一文章，说是使用AVPlayer的AVPlayerItem中stepByCount可以精确到帧的seek
原文如下:“If I use seekToTime to do the scrubbing, it’s pretty slow. What I should use is a method called stepByCount from AVPlayerItem.”

代码例子：


//经测试每[stepByCount:1]一个单位，会seek 一个对应的value值20的CMTime，timescale为player.currentItem.duration.timescale 即CMTimeMake(20, duration.timescale)
[player.currentItem stepByCount:1];//快进
[player.currentItem stepByCount:-1];//快退

简单说一下CMTime

苹果在Core Media框架中定义了CMTime数据类型作为时间的格式，类型定义如下：

typedef struct

CMTimeValue value;
CMTimeScale timescale;
CMTimeFlags flags;
CMTimeEpoch epoch;
} CMTime;
CMTime定义是一个C语言的结构体，其中value是时间值，timescale是时间尺度，flags是标识状态用的，epoch是一个时间时代标识，为了区分当时间戳一样的时候也能区分CMTime的不同，跟循环有关，具体使用场景不详。
CMTime主要是用来播放视频的时候使用的， timescale类型视频fps（每秒的帧数），value除以timescale得到秒数（value/timescale = seconds）
timescale的值网上说法是这样的“You frequently use a timescale of 600, since this is a common multiple of several commonly-used frame-rates: 24 frames per second (fps) for film, 30 fps for NTSC (used for TV in North America and Japan), and 25 fps for PAL (used for TV in Europe). Using a timescale of 600, you can exactly represent any number of frames in these systems”
创建CMTime使用两个常用的方法：
1)CMTimeMake 创建CMTime需要提供value和timescale
CMTimeMake(
int64_t value, /! @param value Initializes the value field of the resulting CMTime. /

CMTime lastframe=CMTimeMake(10, 10);

苹果核心音频(Core Audio)之要点-第一篇

发表于 2016-09-23   |  

本篇文章主要是翻译苹果关于CoreAudio的官方文档，并解读。用来帮助不想看英文文档的你来快速的了解其所阐述的内容。不允许用于商业转载和其它商业模式。(苹果原链接:Core Audio Essentials )

CoreAudio概要

苹果公司设计的针对核心音频的软件接口，采用分层、合作、以任务为中心的方法。阅读本章的前两个部分简要介绍这些接口以及它们如何一起工作。继续阅读了解遍布核心音频的设计原则、使用模式和编程语法。在本章后面的部分向您介绍核心音频如何处理文件、流、录音和回放、以及插件。

API架构层

CoreAudio编程接口设置为三层，如图2-1所示。

图2-1 核心音频的三层接口

最底层包括：
  • I/O Kit:和驱动程序交互
  • Audio HAL:全称是音频硬件抽象层（audio hardware abstraction layer），它提供了一个与设备无关，与驱动程序无关的硬件接口
  • Core MIDI:为MIDI(Musical Instrument Digital Interface,乐器数字接口)流和设备提供软件的抽象
  • Host Time Services:主机时间服务,它提供了访问计算机的时钟

当他们需要尽可能高的实时性能时，Mac应用可以直接使用这些技术。但是许多音频应用程序不访问这一层。事实上，在iOS核心音频提供了实现实时音频使用高级别的接口方式。例如OpenAL，采用直接I/O对于游戏中的实时音频。其是为移动平台适当调整，且相当轻量。

中间层包括用于数据格式转换、读取和写入到磁盘、解析流以及与插件一起工作的服务：
  • Audio Converter Services：让应用程序工作与音频数据格式转换器。
  • Audio File Services：支持从磁盘文件读取数据和写入数据到磁盘文件
  • Audio Unit Services and Audio Processing Graph Services：让应用程序工作与数字信号处理（DSP）的插件，如均衡器和混频器。
  • Audio File Stream Services：创建可以解析流的应用程序，例如用于通过网络连接进行播放的文件。也就是通常所说的流播放
  • Core Audio Clock Services：支持音频和MIDI同步以及时间基准转换。
最高层包括结合了从较低层的功能并精简的接口：
  • Audio Queue Services：让您记录、播放、暂停、循环和同步音频。它采用的编解码器进行必要处理压缩音频格式。
  • AVAudioPlayer class：提供了一个简单的Objective-C接口用于播放和循环音频以及实现快退快进。
  • Extended Audio File Services：整合了Audio File Services 和 Audio Converter services的功能，它给你一个统一的接口，用于读取和写入压缩和非压缩的声音文件。
  • OpenAL：基于开源的OpenAL音效标准实现，它是建立在顶部的系统提供的三维调音台音频单元。所有的应用程序都可以使用OpenAL，虽然它是最适合游戏开发。

框架

Core Audio framework 是以下框架的一个统括，也是一个确实存在的框架

  • Audio Toolbox framework (AudioToolbox.framework)：提供了中间层和最高层的一些服务。iOS中包括Audio Session Services
  • AV Foundation framework (AVFoundation.framework)：在iOS上可用，提供AVAudioPlayer类，一个改进的简单的Objective-C接口用于音频播放
